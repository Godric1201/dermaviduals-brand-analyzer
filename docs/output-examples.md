# Output Examples

This document summarizes the main outputs generated by the AI Recommendation Readiness Diagnosis workflow.

The tool still measures AI answer visibility, competitor mentions, and share of voice. The main executive narrative now uses those benchmark signals to answer a more practical question:

> Why does AI recommend other brands instead of yours, and what evidence should you build first?

---

## Recommendation Readiness Report

The primary Markdown report is designed to diagnose whether the target brand is ready to enter AI recommendation candidate sets for the selected category, market, audience, and use-case context.

For zero-visibility brands, the report uses a First Detection Strategy and includes:

1. Recommendation Readiness Verdict
2. Brand Understanding Summary
3. Who AI Retrieved Instead
4. Why Those Brands Were Retrieved
5. Target vs Retrieved Brand Gap
6. First 3 Evidence Assets to Build
7. Validation Plan
8. Supporting Benchmark Data
9. Methodology / Reliability Notes

The report is benchmark-based. Retrieval roles and target gaps are AI-inferred hypotheses, not verified market facts unless source-grounded research is added separately.

---

## Recommendation Readiness Verdict

The verdict summarizes the most important benchmark interpretation.

Typical fields include:

- Visibility State
- Strategy Mode
- Candidate-set status
- First objective
- Reliability Level
- Short reliability rationale

For zero-visibility brands, the first objective is candidate-set inclusion, not immediate share-of-voice growth.

---

## Brand Understanding Summary

The Brand Understanding Probe helps distinguish basic brand recognition from recommendation retrieval failure.

It may summarize:

- Whether AI appears to understand the brand
- Inferred category alignment
- Inferred market alignment
- Inferred audience alignment
- Missing or uncertain evidence
- Recommended interpretation

Probe results are AI-inferred and require validation. They should not be treated as verified facts about the brand.

---

## Who AI Retrieved Instead

This section identifies visible reference brands that appeared when the target did not.

It may include:

- Retrieved brand names
- Mentions
- Prompts visible
- Share of voice
- Visibility state
- Competitor tier, such as category anchor or unclear reference

Retrieved brands should not automatically be framed as direct competitors to attack. Some may be category anchors, aspirational references, or unclear retrieved references.

---

## Why Those Brands Were Retrieved

The report classifies retrieved brands by benchmark-based retrieval role.

Initial role labels include:

- Planning / consulting authority
- Technical infrastructure provider
- Local market provider
- Trust / premium reference
- Budget / practical option
- Niche specialist
- Comparison anchor
- Unclear retrieved reference

These roles are inferred from benchmark signals such as prompt categories, query types, visibility strength, and market relevance signals. They are not source-grounded competitive claims.

---

## Target vs Retrieved Brand Gap

This section translates retrieved-brand patterns into practical gaps for the target.

For each retrieved brand, the report may explain:

- Observed benchmark signal
- Inferred retrieval role
- Inferred target gap
- Evidence asset implication
- Required validation

The goal is to distinguish observed benchmark data from inference and from recommended action.

---

## First 3 Evidence Assets to Build

The main report prioritizes three evidence assets, rather than a broad generic roadmap.

Each asset includes:

- What to build
- Why it matters
- Target retrieval driver
- Targets / prompt groups
- Validation method

Example asset types:

- Service/category entity page
- Market relevance proof page
- Project proof and trust page
- Alternatives and comparison page
- Technical capability proof page
- Planning and project credibility page

These assets are intended to test whether clearer retrievable evidence improves candidate-set inclusion. They are not a promise of AI mentions or a fixed timeline.

---

## Validation Plan

The validation plan explains how to rerun the benchmark and interpret progress.

It should define:

- What prompt groups to rerun
- What counts as progress
- What does not count as proof
- Why first measurable inclusion matters
- Why no guaranteed timeline should be promised

---

## Supporting Benchmark Data

Detailed benchmark artifacts remain available, but they support the diagnosis rather than dominate the executive narrative.

Supporting data may include:

- Competitive Benchmark
- Trigger-Level Visibility Findings
- Top Brand Winners by Query Type
- Evidence Gap Map
- Prompt-level results
- Detailed score tables
- Raw answer evidence, when included

---

## Benchmark Snapshots

Benchmark snapshots are exported as JSON files and can be compared over time.

They are useful for tracking whether evidence changes are associated with changes in AI recommendation visibility.

Snapshot metadata may include:

- Target brand
- Category
- Market
- Audience
- Competitor set
- Prompt categories
- Prompt count
- Run mode
- Summary metrics
- Diagnostic outputs
- API usage summary
- Optional raw answer evidence

Snapshot comparisons should be interpreted as benchmark signals, not definitive proof of business performance or market preference.

---

## DOCX Executive Report

The tool can also export DOCX executive reports. DOCX formatting may not always match the latest Markdown diagnosis structure immediately; Markdown is the primary format for current Recommendation Readiness report iteration.
